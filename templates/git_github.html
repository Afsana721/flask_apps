CPU Bus Activity

                        

                            This model visualizes the **CPU's central role** in managing the flow of data packets (the white/colored spheres) across the system bus. White packets leave the CPU (Instruction/Data fetch); colored packets return to the CPU (Result/Feedback). This is the engine of all software execution.
                        

                        

                            *Components: RAM, Storage, I/O*
                        

                    

                

                

                

                    

                    

                    

                    

                        
Input → Processing → Output

                        
Input: Your action (key press or mouse click) becomes an electrical signal and is converted into binary (1s/0s).

                        
Processing: The CPU fetches an instruction from RAM, decodes it, and executes it—billions of times per second using the ALU.

                        
Output: Processed data is sent to the monitor (via GPU) or speakers (via sound card) to display or play the final result.

                    The Inner Workings of Git (The .git/ Directory)
                

                

                    

                        Git (Global Information Tracker) is a distributed version control system (DVCS) that operates by storing your entire project history as compressed snapshots inside its internal .git/ folder. This folder is the heart of the repository, containing the essential elements: the **object database**, the **index** (or staging area), all **references** such as branches and tags, and the **HEAD pointer**.
                    

                    

                        Git's architecture is based on three main object types. File content is saved as **blob objects** (Binary Large OBjects). The organization of folders and files is captured by **tree objects**. Finally, each snapshot is recorded as a **commit object** that points to its corresponding tree and its parent commit(s). Every single object in Git is immutable, stored as raw bytes, and uniquely identified by a secure SHA-1 hash.
                    

                    

                        The workflow is tightly managed: when you execute git add, Git converts the file’s content into a blob and registers it in the index. A subsequent git commit then creates the necessary tree and commit objects, finalizing the snapshot and advancing the current branch pointer to the new commit ID. For efficiency, Git optimizes storage using **packfiles**, storing the complete latest version of a file and representing older versions only as compressed differences, or **deltas**.
                    

                    

                        In essence, Git functions as a highly sophisticated, content-addressable mini-filesystem. It uses lightweight references (branches and tags) mapped to commit IDs for navigation, and communicates using a smart protocol to ensure only the absolutely necessary objects are transferred during remote operations.

Core Concepts Explained

                    

                        
                    

                


            

         

          
         

            
Git → GitHub & How It Works Inside


            

                Git (Global Information Tracker) is a distributed version control system (DVCS) that runs locally and stores your project history inside the hidden .git folder. It saves your work as snapshots using internal objects and uses the operating system only for disk, memory and shell access.
            


            

                The .git directory is Git’s brain: it holds the object database, the index (staging area), refs for branches and tags, and the HEAD pointer. When you stage changes, Git stores file content as a blob (Binary Large OBject), tracks folder structure as a tree, and records snapshots as a commit that links to previous commits. Every object is named by a SHA‑1 (Secure Hash Algorithm 1) fingerprint of its raw bytes.
            


            

                Your core loop is simple: add changes to the index, commit to create a snapshot, then push to send that snapshot to a remote. Conflicts happen when two branches change the same lines or one branch deletes what another edits; Git marks the file and pauses so you can manually choose the final content, stage it again, and complete the commit.
            


            

                Older states of the code are always recoverable by their commit ID. You can move to a past commit to inspect it, or pull a specific file from that point back into your current branch so your existing work is not lost. Branches themselves are just lightweight references that point to commit IDs and can be merged so that both histories are preserved.
            


            

                Under the hood, Git behaves like a small content‑addressable database: content → SHA‑1 → stored object. It can pack many objects into compressed packfiles to save space and transfer them efficiently. The classic joke description of Git is “the stupid content tracker” because it simply tracks content by hashes and leaves the meaning and workflow to you.
            


            

                Motto: Gitting your project locally, tracking every change — then GitHub stores and shares it globally.
